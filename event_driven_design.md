
# Event-Driven Architecture: Fitness Tracker ДЗ 6: (Вариант 14)

## 1. Анализ системы (первоначальная версия)

Сначала система казалась достаточно простой.

### Основные сущности:
- User  
- Workout  
- Exercise  

Изначально идея была следующая: пользователь создаёт тренировки и добавляет в них упражнения, а вся логика работает через обычные REST-запросы.

---

## 2. Как менялось понимание системы

В процессе работы стало понятно, что простая монолитная структура не очень подходит, особенно если система будет расширяться.

### Версия 1 (простая архитектура)

Изначально я планировал:

- один сервис
- одна база данных
- стандартные CRUD-операции через REST

Но в процессе стало очевидно, что:

- сложно масштабировать
- логика начинает смешиваться
- трудно добавлять новые функции (например уведомления)

---

### Версия 2 (event-driven архитектура)

После изучения темы event-driven подхода я решил переработать систему:

- добавил RabbitMQ как брокер сообщений  
- разделил систему на сервисы  
- ввёл событийную модель  
- разделил чтение и запись (CQRS)

---

## 3. Команды и события

### Команды (Commands)

| Команда | Описание |
|---|---|
| CreateUser | создание пользователя |
| CreateWorkout | создание тренировки |
| AddExerciseToWorkout | добавление упражнения |
| CreateExercise | добавление упражнения в каталог |

---

### События (Events)

| Событие | Когда появляется |
|---|---|
| UserCreated | после регистрации пользователя |
| WorkoutCreated | после создания тренировки |
| ExerciseAddedToWorkout | при добавлении упражнения |
| WorkoutCompleted | при завершении тренировки |

---

## 4. Итоговая архитектура

```text
User / Workout / Exercise Services (Write side)
                ↓
            RabbitMQ
                ↓
   StatisticsService / NotificationService
                ↓
           Read Model
````

---

## 5. Выбор RabbitMQ

Сначала я рассматривал Kafka, но в итоге отказался от неё.

### Причины:

* сложнее в настройке
* требует больше инфраструктуры
* для учебного проекта избыточна

### Поэтому выбор пал на RabbitMQ, потому что:

* проще запустить через Docker
* удобная маршрутизация через exchanges
* подходит для небольших event-driven систем

---

## 6. CQRS (как я понял в процессе)

Изначально CQRS казался сложной архитектурой.

Но в процессе реализации стало проще понимать:

* Write side → изменяет данные
* Read side → только читает и отдаёт готовые данные

Главная идея — разделение ответственности.

---

## 7. Проблемы при реализации

Во время разработки возникли некоторые сложности:

* RabbitMQ не сразу запускался в Docker → добавил retry-подключение
* появлялись дубликаты сообщений → добавил проверку `event_id`
* синхронизация данных → упростил read model до in-memory структуры

---

## 8. Итог

В результате получилась работающая event-driven система, в которой:

* события передаются через RabbitMQ
* реализовано разделение Write и Read моделей (CQRS)
* есть простая read model для статистики
* система работает асинхронно и расширяемо
